# 프로젝트 규칙 (PROJECT_RULES.md)

> 이 파일은 프로젝트의 핵심 규칙을 담고 있습니다.
> Claude는 코드 작성/수정 시 이 규칙들을 반드시 참조해야 합니다.

---

## 1. 핵심 원칙

### 1.1 코드 품질
- 모든 코드는 **가독성**을 최우선으로 함
- 함수/변수명은 **명확하고 설명적**으로 작성
- 주석은 "왜(why)"를 설명, "무엇(what)"은 코드로 표현
- 중복 코드 금지 - DRY 원칙 준수

### 1.2 아키텍처
- 단일 책임 원칙 (SRP) 준수
- 의존성은 항상 **상위 → 하위** 방향으로
- 순환 의존성 절대 금지
- 레이어 간 직접 접근 금지 (인터페이스 통해서만)

### 1.3 에러 처리
- 모든 외부 호출에 에러 처리 필수
- 사용자에게 보여줄 에러와 로깅용 에러 구분
- 조용히 실패하지 말 것 (silent failure 금지)

## 2. 코딩 컨벤션

### 2.1 네이밍 규칙
```
파일명: kebab-case (예: user-service.ts)
클래스: PascalCase (예: UserService)
함수/메서드: camelCase (예: getUserById)
상수: UPPER_SNAKE_CASE (예: MAX_RETRY_COUNT)
타입/인터페이스: PascalCase + I 접두사 선택 (예: User 또는 IUser)
```

### 2.2 파일 구조
```
src/
├── components/     # UI 컴포넌트
├── services/       # 비즈니스 로직
├── utils/          # 유틸리티 함수
├── types/          # 타입 정의
├── hooks/          # 커스텀 훅 (React)
└── constants/      # 상수 정의
```

### 2.3 임포트 순서
1. 외부 라이브러리
2. 내부 모듈 (절대 경로)
3. 상대 경로 모듈
4. 타입 임포트
5. 스타일/에셋

## 3. 금지 사항

### 3.1 절대 금지
- [ ] `any` 타입 사용 (불가피한 경우 주석으로 사유 명시)
- [ ] `console.log` 프로덕션 코드에 남기기
- [ ] 하드코딩된 비밀키/API 키
- [ ] `eval()` 사용
- [ ] 무한 루프 가능성 있는 코드

### 3.2 권장하지 않음
- [ ] 3단계 이상의 중첩 콜백
- [ ] 100줄 이상의 함수
- [ ] 매직 넘버 (상수로 정의할 것)
- [ ] 암시적 타입 변환에 의존

## 5. Git 규칙

### 5.1 커밋 메시지
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type:**
- feat: 새 기능
- fix: 버그 수정
- docs: 문서 변경
- style: 코드 포맷팅
- refactor: 리팩토링
- test: 테스트 추가/수정
- chore: 빌드/설정 변경

### 5.2 브랜치 네이밍
```
feature/기능명
fix/버그명
hotfix/긴급수정명
```

## 6. 프로젝트 특화 규칙

> 이 섹션은 프로젝트 진행 중 추가되는 특화 규칙을 담습니다.

### 6.1 UI/UX (필수 준수)

**표준 컴포넌트 라이브러리:**
- **ShadCN/UI** (https://github.com/shadcn-ui/ui) 사용 필수
- UI 작업 시 기존 ShadCN 컴포넌트 우선 사용

**컴포넌트 커스터마이징:**
- 모든 커스터마이징 내용을 문서화 필수
- 커스터마이징 시 코드 또는 문서에 "무엇을, 왜 변경했는지" 기록

**강제 규칙:**
```
1. UI 구현 시 → ShadCN/UI 컴포넌트 먼저 확인
2. ShadCN에 적합한 컴포넌트 있음 → 그대로 사용
3. 커스터마이징 필요 시 → 변경 사항 문서화
4. 새 컴포넌트 생성 시 → ShadCN 스타일 가이드 준수
```
